This section highlights the breaking changes in v2 of the BookHub Publisher API. If you use v1, review these changes for a smoother migration.

## Title sorting removed (breaking change)

v2 does not support title sorting in GET all books. If you use title sorting, see [Migrate away from title sorting](/how-to-guides/how-to-guides-migrate-away-from-title-sorting) for the steps to update.

### v1 vs. v2 behavior

The `sort` parameter in GET all books endpoint has changed. In v1, you could sort by `title`. In v2, sorting is supported only by `createdDate`.

In v1, this worked:

```http
GET /v1/books?sort=title&order=asc
```

#### ❌ Invalid request

In v2, this won't work:

```http
GET /v2/books?sort=title&order=asc
```

**Returns a `400` error**

```json
{
  "status": "error",
  "message": "Invalid sort parameter",
  "errors": [{
    "field": "sort",
    "error": "INVALID_VALUE",
    "description": "v2 only supports sorting by createdDate. Use v1 for title sorting."
  }]
}
```

#### ✅ Valid request

In v2, use this instead:

```http
GET /v2/books?sort=createdDate&order=desc
```

### Available sort options in v2

If you use `order` without `sort`, the default `sort=createdDate` applies. Pagination parameters (`page`, `limit`) and filtering (`status`) are unchanged from v1.

The `sort` parameter now accepts **only one value**:

- `createdDate` (default: newest first)

The `order` parameter still works:

- `asc`: Ascending (oldest to newest)
- `desc`: Descending (newest to oldest, default)

## Known issues

The BookHub Publisher API v2 has the following known issues that will be addressed in future versions:

### `validation_context` and `provided_value` use snake_case

The `validation_context` and `provided_value` fields in the error response schema use snake_case rather than the camelCase convention used throughout the rest of the API. This inconsistency will be corrected in a future version. Plan your error handling code accordingly.
